# ПСЦ — Travel Hub CRM · Product Blueprint

> Единый продуктовый документ архитектуры CRM туристической компании.
> Уровень амбиции: Salesforce / HubSpot / Amadeus.
> Документ описывает **целевую архитектуру** и помечает, что уже реализовано в коде, а что — следующий шаг.

## 0. Текущее состояние (аудит кодовой базы)

Стек: один `index.html` (React UMD + `@babel/standalone`, без сборки), модули — `js/*.jsx`, всё цепляется на `window`. Дизайн-токены — в `:root` блоке `index.html`. Библиотека компонентов — `js/ui.jsx`.

**Реализовано (11 фаз):**

**Архитектурный пробел (предмет редизайна):** отсутствует **глобальный shell-слой** — Topbar, Breadcrumbs, глобальный поиск, Notification Center как slide-over, глобальный чат-Drawer. Из-за этого «контекст работы» держится только внутри карточки заказа, а на уровне всего продукта — теряется.

### Связи между модулями (data flow)

```
Лид/Клиент ──► Заказ ──► Услуги (Авиа/ЖД/Отели/...) ──► КП (варианты)
                 │              │                          │
                 │              │                  согласован 1 вариант
                 │              ▼                          ▼
                 │         Поставщик               Оформление (стадия)
                 │                                  ├─► Финансы (операции)
                 │                                  └─► Документы (задачи)
                 ▼                                         │
            Постпродажа ◄──── Возвраты/Обмены ◄────────────┘
                 │
                 ▼
   Уведомления  ◄── события всех модулей (deep-link → объект)
   Чат          ◄── привязан к заказу/клиенту/поставщику
   История      ◄── каждый модуль пишет в audit log заказа
```

Ключевой принцип: **объект-носитель контекста = Заказ**. КП, финансы, документы, возвраты, чат, уведомления — все ссылаются на `order`. Поэтому глобальный чат и уведомления всегда могут открыть нужный заказ через `openOrder(order)`.

**Новый компонент `Topbar` (глобальный, в `AppShell`):**

2. **Глобальный поиск** (`⌘K`) — единое поле: заказы, клиенты, КП, услуги. Раскрывается в поповер с группами результатов и deep-link.

### 2.3 Паттерны страниц (обязательны для всех реестров)

Каждый модуль-реестр использует один скелет (уже частично в коде):

- **Topbar страницы** (`Topbar` из layout.jsx) — заголовок + правые контролы.
- **Панель управления:** поиск · `FilterChip` фильтры · сортировки (`useSort`/`Th`) · переключатель вида.
- **Таблица:** настройка колонок (показать/скрыть, порядок), массовый выбор (чекбоксы) → **bulk-action bar**.
- **Состояния:** `AsyncBlock` / `SkeletonRows` / `EmptyState` (loading / error / empty / ok).
- **Карточка объекта:** открывается внутри страницы (list↔card view) или в Drawer.

## 3. Notification Center (slide-over по колокольчику)

**Цель:** управление вниманием, а не лента сообщений. Сейчас построено как страница (`js/page_notifications.jsx`) — нужно дать **первичный вход через slide-over по 🔔**, переиспользуя тот же `NotificationsCenter`.

**Структура slide-over:**

```
┌─ Уведомления ──────────────────── [⚙] [Прочитать все] [✕]
│ ┌ Требуют внимания · Критические · Непрочитанные · Закреплённые ┐  (stat tabs)
│ Фильтры: [Все][Действия][Непрочит.][Закреплённые]  · источник ▾ · 🔎
├──────────────────────────────────────────────────────────────────
│ 🔴 Новый комментарий в заказе #2024        2 мин   @Менеджер
│    Клиент уточняет дату вылета                    → Открыть заказ
│ 🟡 Изменён статус оплаты — заказ #1542      14 мин
│    Частичная оплата 50%                          → Открыть финансы
│ ⚪ Бронирование подтверждено поставщиком    1 час
│    PNR ABCDEF                                    → Открыть услугу
└──────────────────────────────────────────────────────────────────
```

**Для каждого уведомления:** иконка типа · приоритет (4 уровня `NOTIF_PRIORITY`) · заголовок · описание · время · ответственный · связанный объект · статус прочтения · pin/read/dismiss · **кнопка deep-link**.

**Фильтры по источнику (`NOTIF_SOURCE`):** Заказы · Платежи · Чаты · Бронирования · Системные.

**Маршрутизация действий** — `notifGo(n, onNavigate, onOpenOrder)`: `link.type==='order'` → `openOrder`; иначе `navigate('finance'|'documents'|'returns'|'offers')`.

**Целевое улучшение:** deep-link на **конкретную карточку записи**, а не только в раздел (сейчас плумбинг межмодульного intent не разведён глубже заказа).

Данные: `NOTIFICATIONS`, `NOTIF_PRIORITY`, `NOTIF_PRIO_RANK`, `NOTIF_SOURCE`, `NOTIF_SETTINGS` (`js/data.jsx`).

## 4. Global Chat Drawer (slide-over поверх любой страницы)

**Концепция (ядро редизайна):** чат — НЕ отдельный экран. Это **Drawer справа**, открывается из Topbar 💬 или из карточки заказа, **не закрывая текущую страницу**. Пользователь продолжает видеть заказ/реестр под полупрозрачным оверлеем.

Компонент `ChatThread` уже переиспользуем (`js/page_chats.jsx`) и встроен во вкладку «Чат» карточки заказа. Нужно поднять его в **глобальный Drawer на уровне `AppShell`**, с состоянием «активный тред = тред текущего контекста».

**Структура Drawer:**

```
┌─ Чат · Заказ №1542 ───────────────────────── [🔎] [✕]
│ Иванов И.И. · Тип: Бронирование · Ответственный: Петров А.
├ [ Клиент ② ] [ Поставщик ] [ Внутренние ① ]      ← каналы (CHAT_CHANNELS)
├──────────────────────────────────────────────────────
│  ┌ системное событие (статус изменён) ┐  (центр. pill)
│  Менеджер · 14:02                          ✓✓ прочитано
│   Подтвердите, пожалуйста, даты            📎 voucher.pdf
│                          Клиент · 14:05 →
│                          Да, всё верно
│  @Петров проверьте тариф   (внутр. канал — янтарный фон)
├──────────────────────────────────────────────────────
│ [📎] [ Сообщение…  @упоминание ]                 [➤]
└──────────────────────────────────────────────────────
```

**Заголовок:** № заказа · имя клиента · тип обращения · ответственный менеджер.

**3 канала (роли-разделение):** Клиент · Поставщик · Внутренние (только сотрудники, янтарный фон). Бейджи непрочитанного на вкладках каналов.

**Возможности:** история сообщений · вложения (file chip) · @mentions (подсветка `chatText`) · статус прочтения (✓✓) · системные события (центрированный pill) · поиск по сообщениям · фильтр истории · внутренние комментарии (отдельный канал).

**Поведение контекста:**
- Открыт заказ → 💬 открывает тред этого заказа (`getThreadForOrder(order)`).
- Открыт реестр → 💬 открывает список тредов + выбранный тред.
- Внутри Drawer — чип «Открыть заказ» (`onOpenOrder`) для перехода к объекту.

Данные: `CHAT_THREADS` (`{channels:{client,supplier,internal}}`), `CHAT_CHANNELS` (`js/data.jsx`).

## 5. Модуль Коммерческих предложений (КП)

**Реализован** (`js/page_offers.jsx`, Phase 3). Это один из ключевых модулей и он **мультисервисный**.

**Модель:** `Proposal` принадлежит заказу и содержит несколько `Variant`; каждый вариант = список `items {kind,title,sub,cost,fee}`. Клиент согласует **ОДИН** вариант → статус `Согласовано` + `approvedVariant`.

**Поддерживаемые типы в одном КП:** ✈ Авиа · 🚆 ЖД · 🏨 Отели · 🚗 Трансферы · 🛡 Страхование · 🎫 Экскурсии · 📄 Визы · ⭐ Доп. услуги.

**Экраны:**

| # | Экран | Где |
|---|---|---|
| 1 | Список всех КП | `OffersPage`/`OffersRegistry` (route `offers`): поиск/фильтр/сорт/статус/превью-модал/«Перейти в заказ» |
| 2 | Создание нового КП | в конструкторе (variant tabs) |
| 3 | Конструктор КП | variant tabs (add/duplicate/delete/rename) · редактируемая таблица услуг (inline cost/fee) · reorder · итоги по варианту · валюта/срок/участники |
| 4 | Просмотр КП | `KPPreviewDoc` — брендированный клиентский документ |
| 5 | История изменений | `KPHistoryDrawer` (timeline по proposal) |
| 6 | Шаблоны КП | *целевое* — сохранение варианта как шаблона (расширение) |
| 7 | Отправка клиенту | действие со сменой статуса → `Отправлено` |
| 8 | Статусы (`KP_STATUS_FLOW`, 7 шт.) | Черновик · Отправлено · Просмотрено · Принято · Отклонено · Требует изменений · Согласовано |

**Связь с заказом:** `KPModule({order, services, participants, onApprove})` встроен во вкладку «Ком.предложения» карточки заказа. При согласовании → заказ переходит в стадию «Оформляется», создаются фин-записи и задачи на документы (toast), появляется зелёный bulk-action bar: **Забронировать авиа · Сформировать документы · Открыть финансы**.

## 6. Спецификации модулей (по экранам)

Формат каждого модуля: **Цель · Сценарий · Структура · Блоки/Таблицы/Фильтры/Формы/Кнопки · Связи · Роли.**

### 6.1 Dashboard (`js/page_dashboard.jsx`)
- **Цель:** утренняя точка входа — что требует внимания сегодня.
- **Структура:** KPI-карточки (заказы/выручка/дедлайны) → `ActivityFeed` (топ событий по приоритету, ссылка «Все уведомления») → Мои задачи/дедлайны → Быстрые действия (`onAddOrder`, навигация).
- **Связи:** карточки → `openOrder`; быстрые действия → `navigate`/`createOrder`.
- **Роли:** Руководитель видит сводные KPI команды; Оператор — только свои заказы/задачи.

### 6.2 Заказы — Реестр (`js/page_orders.jsx` → `OrdersPage`)
- **Цель:** найти/создать заказ, перейти в работу.
- **Таблица:** №, клиент, маршрут, даты, статус, оператор, сумма, оплата.
- **Фильтры:** статус · оператор · даты · тип услуги · оплата. **Поиск.** **Сорт** (`useSort`). **Массовые действия** (назначить оператора, сменить статус, экспорт). **Настройка колонок.**
- **Формы:** «Новый заказ» (`intent:create`).
- **Связи:** строка → Карточка заказа; `intent` из `app.jsx` (`open`/`create`).

### 6.3 Заказы — Карточка (`js/page_order_card.jsx` → `OrderCard`) ◀ главный workspace
- **Header:** № · редактируемый статус · `OrderStageBar` (6 стадий) · оператор · клиент · даты поездки · quick actions.
- **Sticky right panel:** фин-сводка (к оплате / оплачено / задолженность / комиссия) + дедлайны.
- **10 вкладок:**
  - **Общая** — резюме заказа.
  - **Клиенты** — заказчик (→ карточка клиента).
  - **Участники** — пассажиры (`PassengerDrawer`, `PassportModal`).
  - **Маршрут** — таймлайн поездки.
  - **Услуги** — мультитиповые строки; авиа → встроенный `FlightCard`; «Добавить услугу» → `FlightSearch/Results` (авиа) или `ServiceAddFlow(routeKey)` (ЖД/отели/…); добавление пушит в `services`, фин-сводка пересчитывается.
  - **Ком.предложения** — `KPModule` (§5).
  - **Документы** — `DocCenter` со `scopeOrder`.
  - **Финансы** — `FinanceRegistry` со `scopeOrder`.
  - **Постпродажа** — `ReturnsModule` (compact): активные возвраты/обмены/завершённые.
  - **История** — audit log заказа.
- **Связи:** пишет во все модули; чат — встроенный `ChatThread` (целевое: глобальный Drawer §4).

### 6.4 Услуги (Авиа + 5 модулей)
- **Авиа (`js/page_flights.jsx`):** реестр → поиск → выдача (фильтр+сорт+сравнение) → карточка услуги. Экспортирует `FlightSearch/FlightResults/FlightCard/AirlineLogo`.
- **ЖД/Отели/Трансферы/Автобусы/Туры (`js/page_services.jsx`):** один framework `ServiceFlow({routeKey})`, конфиг `SVC_CFG[routeKey]` (поля `SvcField`: text/date/daterange/stepper/select). Данные `SVC_DATA`.
- **Связи:** офер добавляется в заказ как услуга (`ServiceAddFlow` → `addSvcOffer`).

### 6.5 Клиенты / Компании (`js/page_people.jsx`)
- **ClientCard:** контакты · KPI · заказы клиента (`ordersOf` → `openOrder`) · документы.
- **CompanyCard:** реквизиты · банк/договор · контактные лица · заказы компании.

### 6.6 Поставщики (`js/page_suppliers.jsx`) — реестр + карточка, intent из app.jsx.

### 6.7 Финансы + Оформление + Документы (`js/page_fulfillment.jsx`)

## 7. Система ролей

`ROLES` и матрица `PERMISSIONS` — `js/data.jsx`, редактор — `RolesTab`.

| Роль | Доступные разделы | Ключевые права | Особенности UI |
|---|---|---|---|
| **Администратор** | всё | управление пользователями/ролями/API; колонка прав заблокирована «на себя» | видит Настройки целиком, журнал аудита всех |
| **Руководитель** | всё кроме системных настроек | сводная аналитика, переназначение заказов, утверждение скидок | Dashboard = KPI команды; видит чужие заказы |
| **Менеджер** | Заказы, Услуги, КП, Клиенты, Чаты, Уведомления | вести свои заказы, создавать КП, общаться с клиентом | видит только свои заказы; внутренний канал чата |
| **Оператор** | Услуги, Бронирования, Поставщики, часть Заказа | бронировать у поставщика, менять статусы услуг | канал «Поставщик» в чате; без финансов |
| **Бухгалтер** | Финансы, Документы, Оформление, Возвраты | проводить оплаты/возвраты/комиссии, формировать документы | финансовые вкладки заказа; без редактирования услуг |

**Принцип адаптации интерфейса:** скрытие пунктов Sidebar по правам · disabled-кнопки действий вне прав · вкладки карточки заказа фильтруются по роли · каналы чата по роли.

## 8. Пользовательские сценарии (User Flows)

**Сценарий 1 — полный жизненный цикл:**
```
Клиент обратился → [Quick Create] Лид/Заказ → добавить Участников
 → добавить Услуги (Авиа/ЖД/Отели…) → собрать КП (несколько вариантов)
 → Отправить клиенту → клиент Просмотрел → Согласовал 1 вариант
 → заказ → стадия «Оформляется» → авто фин-операции + задачи на документы
 → Оплата (Финансы) → Бронирование у поставщика → Документы клиенту
 → Поездка → Завершение.  (история пишется на каждом шаге)
```

**Сценарий 2 — реакция на событие:**
```
🔔 Уведомление (slide-over) → deep-link → объект открыт в контексте
 → действие → система фиксирует в Истории → уведомление помечено прочитанным.
```

**Сценарий 3 — работа с чатом без потери контекста:**
```
Оператор в карточке заказа №1542 → жмёт 💬 (Topbar)
 → Chat Drawer открывается СПРАВА, заказ виден слева
 → переписка (канал Клиент/Поставщик/Внутренние), вложения, @упоминания
 → закрывает Drawer → продолжает работу с тем же заказом, контекст сохранён.
```

**Сценарий 4 — постпродажа:**
```
Клиент просит возврат → в заказе вкладка «Постпродажа» → NewReturnModal
 → тип + услуга → авто фин-оценка → ProcessFlow → завершение
 → создаётся фин-операция «Возврат», задолженность обновлена, документы авто-привязаны.
```
