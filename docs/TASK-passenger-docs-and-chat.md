# Задача 1 — Документы группируются по пассажиру

## Что сказал клиент
> У каждого пассажира свой набор документов для быстрой навигации. Суть в том, что у каждого
> пассажира свой набор документов по командировке, не считая бухгалтерских. Градация по
> пассажирам сделана для более удобного доступа/поиска нужных доков.

Референс — скриншот «Sikora: просмотр заказа», вкладка «Состав поездки»:
- Услуги сгруппированы по дате (`05.02.2026, четверг`).
- Внутри даты — карточка рейса (`Шэньян → Москва`, авиакомпания, цена).
- **Внутри карточки рейса — список пассажиров, и у каждого пассажира отдельно прикреплён
  его билет** (Диденко Егор Викторович → билет 9996389894032, Уткин Федор Денисович → билет
  9996389894033, Диденко Захар Егорович → билет 9996389894034).
- Бухгалтерские документы (акты, счета, договоры) сюда не входят — они отдельно.

## Текущее состояние в коде
- `js/page_order_card.jsx`, вкладка «Документы» (`case 'documents'`) рендерит `<DocCenter scopeOrder={order.no} />`
  из `js/page_fulfillment.jsx:275`. `DocCenter` — это **плоская таблица документов**, сгруппированная
  по типу документа (вкладки «Все / Билеты и квитанции / Ваучеры и полисы / Счета и акты / Договоры /
  Паспорта / Требуют действия»), у каждой строки есть `d.participant`, но группировки по пассажиру нет.
- Есть устаревшая функция `TabDocuments` (`js/page_order_card.jsx:785-814`) — она не используется
  (в `OrderCard` для таба `documents` вызывается `DocCenter`, а не `TabDocuments`). Она хардкодит одного
  пассажира («Нуралиев Данияр») и список типов документов без реальных данных. **Эту функцию нужно
  либо удалить, либо переиспользовать как основу для новой группировки** — решить по месту, в идеале
  удалить, чтобы не оставлять мёртвый код.
- Сервисы заказа (`ORDER_SERVICES`, см. `js/data.js` или аналогичный файл с моками — найти grep'ом
  `ORDER_SERVICES`) сейчас не хранят информацию «какой документ привязан к какому пассажиру в рамках
  конкретной услуги». Участники заказа — `ORDER_PARTICIPANTS` / `GROUP_PAX` (используются в
  `TabParticipants`, `js/page_order_card.jsx:351`).
- Документы (`DOCS2`, мок-данные для `DocCenter`) уже имеют поле `participant`, но это просто текстовое
  поле для отображения в таблице, не структура «услуга → пассажир → документ».

## Что нужно сделать
1. **Добавить вкладке «Документы» в карточке заказа (`TabDocuments`/раздел внутри `DocCenter` со
   `scopeOrder`) режим группировки по пассажиру**, который воспроизводит логику со скриншота:
   - Верхний уровень: список пассажиров заказа (из `ORDER_PARTICIPANTS`/`GROUP_PAX` для текущего
     заказа).
   - Для каждого пассажира — список его документов **по поездке** (билеты, посадочные, ваучеры,
     страховки, визы), полученных через фильтр `DOCS2` по `d.order === order.no && d.participant === p.name`,
     **исключая** бухгалтерские типы (`Счёт`, `Акт`, `Договор` — см. `DOC_KIND`/`DOC_STATUS2` в
     `js/page_fulfillment.jsx` или соответствующем файле с константами).
   - Бухгалтерские документы показываются отдельным блоком ниже/на отдельной вкладке («Документы по
     заказу» / «Бухгалтерия»), не привязанным к конкретному пассажиру — как в текущем `DocCenter`,
     просто отфильтровать по типу.
2. **Переключатель вида**: добавить тумблер/вкладки «По пассажирам» / «По типу документа», чтобы
   старое поведение `DocCenter` (группировка по типу) не потерялось — клиенту может быть нужен и тот,
   и этот разрез. Реализовать как два режима рендера внутри `DocCenter` (проп `scopeOrder` уже есть,
   добавить локальный `view` state: `'byType' | 'byPassenger'`), кнопки переключения сверху таблицы.
3. **Карточка пассажира внутри группировки** должна по клику открывать тот же `DocCard`
   (`js/page_fulfillment.jsx:219`), что и сейчас — просто фильтрация перед рендером меняется, drawer
   остаётся прежним.
4. Если у пассажира нет ни одного документа — показать `EmptyState` с подсказкой «Документы пассажира
   ещё не загружены» и кнопкой загрузки (как в остальных частях прототипа, см. `EmptyState` usage
   паттерн в `TabParticipants`).
5. Если нужно — добавить мок-данные (несколько документов на разных пассажиров в одном заказе), чтобы
   группировка была наглядна на демо-данных (сейчас в `DOCS2`, вероятно, документы привязаны к одному
   участнику на заказ — проверить и расширить при необходимости).

## Что НЕ делать
- Не трогать раздел «Финансы» — бухгалтерские документы (счета, акты, договоры) сознательно
  показываются отдельно от пассажирских, как и просил клиент.
- Не переделывать модель данных глобально (`DOCS2`) кардинально — достаточно фильтрации существующих
  полей `order`/`participant`/`type`.

# Задача 2 — Новый дизайн модуля «Чаты»

## Что сказал клиент
> Чат. Слева навигационное меню чата по всем заказам. Это черновой набросок. В самом чате может
> проходить взаимосвязь с поставщиками услуг, внутренний обмен сообщениями и т.д.

Референс — скриншот «ПСЦ Travel Hub», экран «Чат по заказу»:
1. **Левая колонка — навигация по всем чатам**, не привязанная к одному заказу:
   - Поиск по чатам/участникам/услугам/сообщениям.
   - Переключатель «По типу» / «По услуге».
   - Фильтры по типу чата с счётчиками: «Все чаты» (18), «Клиент» (3), «Операторы» (4),
     «Поставщики» (7), «Локальные поставщики» (2), «События» (2).
   - Список чатов, сортировка («Сначала новые»), у каждого пункта — название, бейдж канала связи
     (MAX / Email / API / Поставщик / Система), превью последнего сообщения, счётчик непрочитанных.
2. **Центр — окно переписки конкретного чата**:
   - Заголовок: номер заказа (`Заказ №51188`), статус (`В работе`), клиент (`ООО "Грид Лимитед"`),
     канал (`MAX`), оператор (`Даниель`), статус подключения канала (`Подключено`), счётчик участников.
   - Кнопки сверху: «Создать задачу», «Закрепить чат», меню «…», закрыть.
   - Лента сообщений: обычные сообщения операторов/клиентов **и системные сообщения**
     («Получен ответ от поставщика Air Astana по запросу №34251. Статус: Подтверждено» — с кнопкой
     «Открыть ответ»).
   - Композер: кнопка «Привязать к услуге» — выбранная услуга показывается как чип
     (`Авиаперелёт FRU → IST ×`), вкладки «Сообщение» / «Внутренний комментарий» (закрытый чат с
     иконкой замка — аналог текущего канала `internal`).
3. **Правая колонка — информация о чате**:
   - «Связано с услугой»: карточка услуги (`Авиаперелёт FRU → IST`, дата/рейс, статус
     `Подтверждено`, кнопка «Открыть услугу»).
   - Блок «Информация о чате»: Тип чата (Клиентский/Поставщик/Внутренний), Канал связи, статус
     подключения, дата создания, ответственный оператор.
   - Блок «Участники» с аватарками + «Показать всех».
   - Блок «Связанные услуги» (может быть несколько, не только одна).
   - Блок «Быстрые действия»: Создать задачу / Отправить email / Открыть карточку заказа /
     История изменений.

## Текущее состояние в коде
- `js/page_chats.jsx` — весь модуль чатов:
  - `getThreadForOrder(order)` (строка 17) даёт **один тред на один заказ** с тремя
    каналами `client` / `supplier` / `internal` (`CHAT_CHANNELS`, видимо в файле с константами).
    Это не модель «много независимых чатов разных типов, привязанных к заказу и/или к услуге» —
    нужно расширять модель.
  - `ChatThread` (строка 23) — переключение между 3 каналами как табы внутри одного треда. В новом
    дизайне это, по сути, разные **отдельные чаты** в списке слева (вкладка «Клиент» = чаты с
    клиентами, «Поставщики» = чаты с поставщиками, и т.д.), а не вкладки внутри одного треда.
    Канал «Внутренний» (`internal`) остаётся как отдельная вкладка композера в самом чате
    («Сообщение» / «Внутренний комментарий» в референсе) — это единственное, что логично оставить
    табом внутри чата, а не отдельным чатом в списке.
  - `ChatsPage` (строка 103) — список слева, без фильтров по типу, без счётчиков по категориям, без
    бейджа канала связи на пункте списка, без правой колонки с информацией о чате.
  - Привязки чата к конкретной **услуге** заказа (а не только к заказу целиком) сейчас нет — чат
    привязан только к `order.no`.
  - Кнопок «Создать задачу» / «Закрепить чат» / быстрых действий нет.
- `OrderCard` (`js/page_order_card.jsx:1039`) открывает чат заказа кнопкой «Чат» — эта точка входа
  должна продолжать работать, но открывать чат уже в новой структуре (правильный тред в общем списке).

## Что нужно сделать
1. **Расширить модель чатов** (там, где определены `CHAT_THREADS`/`CHAT_CHANNELS` — найти grep'ом
   `CHAT_THREADS` в файлах данных):
   - У треда должен быть `type`: `'client' | 'operator' | 'supplier' | 'local_supplier' | 'system'`.
   - У треда должен быть `channel`: `'MAX' | 'Email' | 'API' | 'Система' | ...` — для бейджа в списке
     и в заголовке.
   - У треда — опциональная привязка `relatedService` (id услуги в заказе) или список
     `relatedServices`, плюс `participants` (список людей чата, не только клиент/оператор), `pinned`
     (для «Закрепить чат»), `connectionStatus` (`'Подключено' | 'Отключено'`), `createdAt`,
     `responsibleOperator`.
   - Системные сообщения (`from: 'system'`) уже поддержаны в `ChatThread` (строка 71) — расширить,
     чтобы у системного сообщения можно было показать кнопку действия («Открыть ответ»), которая
     открывает связанную услугу/запрос.
2. **Левая навигация** (новый компонент, например `ChatsNav`, либо переработать список внутри
   `ChatsPage`):
   - Тумблер «По типу» / «По услуге» — второй режим группирует список чатов по услуге заказа
     (`relatedService`), а не по типу участника.
   - Вкладки/фильтры по `type` с количеством чатов каждого типа (считать через `.filter`).
   - В каждой строке списка — аватар/иконка, название, бейдж канала (`channel`), превью последнего
     сообщения (переиспользовать `lastMessage`), счётчик непрочитанных.
3. **Окно переписки** — адаптировать `ChatThread`:
   - Убрать переключение `client`/`supplier` как вкладки **внутри** треда (теперь это разные треды в
     списке слева); оставить только переключатель «Сообщение» / «Внутренний комментарий»
     (`internal`) в области композера, как два под-канала одного чата.
   - В заголовке показать: номер заказа + статус заказа, клиента/контрагента, канал связи и статус
     подключения, ответственного оператора, счётчик участников.
   - Добавить кнопки «Создать задачу» и «Закрепить чат» (`pinned`) в заголовок треда.
   - В композере добавить кнопку «Привязать к услуге» — открывает выбор услуги из заказа
     (переиспользовать список услуг заказа, например через `ActionMenu`, аналогично выбору
     упоминания `@`, строка 91), выбранная услуга отображается как чип над полем ввода с крестиком
     для отвязки.
4. **Правая колонка** (новый компонент, например `ChatInfoPanel`):
   - Блок «Связано с услугой» — карточка(и) связанной услуги с статусом и кнопкой «Открыть услугу»
     (переход в карточку заказа на вкладку «Услуги», к конкретной услуге — можно через `onOpenOrder`
     с доп. параметром `tab: 'services'`).
   - Блок «Информация о чате»: тип, канал, статус подключения, дата создания, ответственный.
   - Блок «Участники» с аватарками (переиспользовать `Avatar`) + «Показать всех» (открыть
     drawer/modal со списком).
   - Блок «Быстрые действия»: Создать задачу / Отправить email / Открыть карточку заказа / История
     изменений — каждая кнопка может быть просто `toast(...)`-заглушкой, кроме «Открыть карточку
     заказа», которая должна реально работать через существующий `onOpenOrder`.
5. **Точка входа из карточки заказа** (`OrderCard`, строка ~1039, кнопка «Чат») должна продолжать
   открывать корректный чат — если для заказа несколько тредов разных типов, открывать тред с
   `type: 'client'` по умолчанию либо последний активный.
6. Обновить мок-данные `CHAT_THREADS`, чтобы на демо было несколько чатов разных типов на один
   заказ (клиентский + чат с поставщиком + системные события), с реалистичными `channel`,
   `relatedService`, `participants` — чтобы новый UI было на чём проверить.

## Что НЕ делать
- Не реализовывать реальную интеграцию с MAX/Email/API — это прототип, статусы канала и сообщения
  остаются мок-данными, как и сейчас.
- Не убирать существующий канал «Внутренний комментарий» — он явно есть и на новом скриншоте, просто
  меняется его место (под-вкладка композера, а не отдельный чат в списке).

# Порядок работы
1. Сначала Задача 1 (документы по пассажирам) — она локальная, в пределах `DocCenter`/`TabDocuments`,
   риск регрессии низкий.
2. Затем Задача 2 (чаты) — она больше по объёму (новая структура данных, минимум 2 новых компонента),
   делать отдельным шагом и проверять, что точка входа «Чат» из карточки заказа не ломается.
3. После каждой задачи — открыть прототип в браузере (`run` skill / dev server проекта) и пройти
   golden path: открыть заказ → вкладка «Документы» → переключить «По пассажирам»; открыть «Чаты» →
   отфильтровать по типу → открыть чат → привязать к услуге → отправить сообщение.
