# Промпт для Claude: исправление замечаний UX-аудита CRM

Это техническое задание по результатам ручного UX-аудита (реальный прогон сценариев в браузере). Используй этот файл как промпт — пройди пункты по порядку приоритета (Critical → High → Medium → Low), для каждого внеси точечное изменение, не трогая остальное.

## Жёсткие ограничения

- **Не меняй бизнес-логику и данные** (`js/data.jsx`), только UX/UI-организацию экранов и поведение конкретных контролов.
- **Не переписывай архитектуру заново.** Drawer/Modal/sticky/accordion-система уже реализована (`js/ui.jsx`: `Drawer`, `Modal`, `ConfirmDialog`, `Tabs`, `ActionMenu` и т.д.; `js/order_extras.jsx`: `CollapseSection`). Используй существующие примитивы, не создавай новые без необходимости.
- **Не плодите дублирующиеся страницы.** Если для починки нужен предпросмотр/подтверждение — используй `Modal`/`ConfirmDialog`/`Drawer`, а не новый route.
- После каждого пункта (или логической группы пунктов) **проверь в браузере** (`python3 -m http.server` + Playwright/скриншот или ручной клик-тест), что:
  - нет ошибок в консоли;
  - сценарий до и после изменения проходится так, как описано в «Ожидаемый результат»;
  - ничего из соседних экранов не сломалось (особенно `OrderCard`, т.к. в нём сходится много sub-flow).
- Коммить можно логическими группами (например, все Critical — один коммит), но не нужно молчать — после каждой группы дай краткий отчёт что изменено и как проверено.

## CRITICAL

### C1. Контент мастера бронирования уходит за экран на всех 7 шагах
**Файл:** `js/page_booking.jsx`, компонент `BookingWizard`.
**Проблема:** при открытии мастера («Начать бронирование») над `bw-top`/`bw-stepper`/`content()` остаётся полная шапка `OrderCard` (`oc-head`: статус, маршрут, facts, пилюли услуг, stage-bar на 6 этапов, 5 кнопок действий) — итого ~700px над мастером. На viewport 1440×900 контент шага (список услуг, сравнение тарифов, чек-лист, итоговый экран) не виден ни на одном из 7 шагов без скролла.
**Ожидаемый результат:** когда `svcView === 'booking'` (см. `renderServicesArea` в `js/page_order_card.jsx`), полная шапка заказа (`oc-head`) должна сворачиваться до компактной строки (номер заказа + статус + маршрут одной строкой, без stage-bar заказа и без 5 кнопок действий — они избыточны во время мастера) **или** не рендериться вовсе, оставляя только `BackRow`/`bw-top`/`bw-stepper`. Главное условие приёмки: на старте каждого из шагов 0–6 в вьюпорте 1440×900 видна хотя бы часть актуального `content()` шага без скролла.
**Как делать:** в `OrderCard` (`page_order_card.jsx`), там где рендерится `oc-head` — оберни в условие, скрывающее его (или показывающее компактную версию) когда `svcView === 'booking'` (уже есть похожее условие `!['avia-picker', 'booking'].includes(svcView)` для `oc-tabs`/`oc-aside` — расширь его на сам `oc-head`, оставив минимальную «шапку-крошку»).

### C2. Кнопка «Редактировать данные» в мастере бронирования на деле закрывает мастер
**Файл:** `js/page_booking.jsx`, `BookingWizard`, верх компонента (`bw-top`).
**Проблема:** `<Button ... icon="edit" onClick={onClose}>Редактировать данные</Button>` — подпись обещает редактирование, а реально вызывает `onClose` (выход из мастера без подтверждения и без сохранения).
**Ожидаемый результат:** либо убрать эту кнопку из `bw-top` совсем (она дублирует кнопку с тем же текстом в `oc-head`), либо переименовать в «Закрыть» / «Выйти из бронирования» и оборачивать в `ConfirmDialog`, если пользователь уже продвинулся дальше шага 0 (чтобы не терять прогресс случайно).
**Приёмка:** подпись кнопки соответствует её реальному действию; если шаг > 0, выход подтверждается диалогом.

### C3. Глобальный поиск услуг (Гостиницы/Трансферы через сайдбар) не привязан к заказу
**Файл:** `js/page_services.jsx`, компонент `ServiceFlow`.
**Проблема:** при заходе через сайдбар «Услуги → Гостиницы/Трансферы» форма поиска не запрашивает заказ/клиента. После выбора предложения (`onSelect`) показывается toast «Услуга добавлена в заказ» — непонятно, в какой.
**Ожидаемый результат:** один из двух вариантов (выбери проще в реализации):
1. Убрать обманчивый toast «добавлена в заказ» из `ServiceFlow.runSearch`/`onSelect`-флоу для не-привязанного к заказу контекста — заменить на нейтральный («Предложение сохранено») и явно показать, что это просто реестр/поиск без привязки, а реальное добавление в заказ происходит только из `TabServices` внутри `OrderCard` (`onAddType` → `ServiceAddFlow`).
2. Либо добавить в `ServiceFlow` шаг выбора заказа (`SearchBox`/`Select` по списку открытых заказов) перед `onSelect`, и только после этого показывать «добавлена в заказ № …».
**Приёмка:** пользователь не видит сообщений про привязку к заказу там, где привязки физически нет.

### C4. В окне документов пассажира видно только 2 из N участников
**Файл:** `js/order_extras.jsx`, компонент `PassportModal`.
**Проблема:** `const pax = [{ name: passenger || '...' }, { name: '...' }]` — список пассажиров в модалке всегда хардкодные 2 человека, независимо от реального состава участников заказа (`ORDER_PARTICIPANTS` / `GROUP_PAX`, которые могут быть 20+).
**Ожидаемый результат:** `PassportModal` должен принимать полный список участников заказа (передавать `participants`/`list` пропом из `OrderCard`/`TabParticipants`, аналогично тому, как уже передаётся `passenger`), и левая колонка выбора пассажира (`pax.map`) должна поддерживать длинный список — со скроллом внутри своего блока (не растягивая всю модалку), и/или поиском по ФИО, если участников больше ~8.
**Приёмка:** для группового заказа на 20 человек можно открыть документы любого из 20, а не только первых двух.

### C5. Кнопки-редактирования в обзоре заказа ничего не делают, но выглядят как успех
**Файл:** `js/page_order_card.jsx`, компоненты `TabOverview`, `TabClients`.
**Проблема:** иконки `edit` рядом с «Основная информация» / «Параметры заказа» / у блока клиента вызывают `toast('Редактирование', 'info')` или `toast('Карточка клиента', 'info')` — показывают успех, ничего не меняя.
**Ожидаемый результат:** либо открыть реальный `Drawer` с формой редактирования соответствующего блока (минимально — переиспользовать паттерн `TripEditModal` как референс: те же поля, что выведены в `kv-stack`, в форме `Field`/`Input`/`Select`), либо, если редактирование пока вне скоупа, заменить иконку на неактивную/убрать, чтобы не создавать иллюзию действия.
**Приёмка:** нажатие на «редактировать» либо открывает форму, которая реально меняет отображаемые данные, либо контрол отсутствует/задизейблен — никаких фальшивых тостов об успехе.

## HIGH

### H1. Дублирующийся выбор мест: шаг 3 «Места и тарифы» и шаг 4 «Допуслуги → Места»
**Файл:** `js/page_avia_picker.jsx`, `AviaPicker` (шаг 3 — `SeatSelector`, шаг 4 — `ExtrasTabs` вкладка «Места»).
**Ожидаемый результат:** оставить выбор мест только в одном месте. Рекомендация: убрать шаг 3 как отдельный аккордеон-шаг и оставить выбор мест только внутри `ExtrasTabs` (шаг 4) — тогда нумерация шагов сокращается до 4 вместо 5, и нет конфликта состояния `extras.seats`.
**Приёмка:** в интерфейсе ровно один контрол выбора мест, использующий единый стейт.

### H2. Невидимая конвертация валют в итоге сценария авиа-пикера
**Файл:** `js/page_avia_picker.jsx`, функция `apply`/итог `grand`, плюс `ServiceList`/`ap-sc-total` (использует `s.sum * 90` как хардкод курса).
**Ожидаемый результат:** там, где сумма по сценарию складывает позиции в разных валютах в один рублёвый итог — показать рядом курс конвертации (например, подпись `≈ 90 ₽/$` под итоговой суммой) либо показывать итог по каждой валюте отдельно, без скрытого пересчёта.
**Приёмка:** пользователь видит, почему итоговая сумма в рублях больше суммы исходных позиций в долларах — курс не скрыт.

### H3. Два несинхронизированных прогресс-индикатора одновременно на экране
**Файлы:** `js/page_fulfillment.jsx` (`OrderStageBar`, этапы из `ORDER_STAGES`) и `js/page_booking.jsx` (`BwStepper`, `STEPS` мастера).
**Ожидаемый результат:** когда открыт `BookingWizard` внутри заказа, stage-bar заказа (`OrderStageBar`) не должен показываться одновременно со степпером мастера (это естественно решится после фикса C1, если `oc-head` со stage-bar скрывается на время мастера) — отдельно проверь, что после выхода из мастера оба индикатора синхронизированы по смыслу (например, завершение мастера = заказ переходит в стадию «Оплачен», что уже работает в коде через `onComplete`, просто визуально не должно дублироваться во время самого мастера).
**Приёмка:** в любой момент времени на экране виден только один прогресс-индикатор, релевантный текущему контексту.

### H4. Дублирующаяся кнопка «Редактировать данные» (шапка заказа + шапка мастера)
**Файлы:** `js/page_order_card.jsx` (`TripSummaryBar`, кнопка «Редактировать данные» → `onEdit`/`setTripEditOpen`) и `js/page_booking.jsx` (`bw-top`, та же подпись → `onClose`).
**Ожидаемый результат:** решается автоматически после C1 (шапка заказа скрыта во время мастера) — оставить кнопку с таким текстом только там, где она действительно открывает `TripEditModal`.
**Приёмка:** подпись «Редактировать данные» встречается на экране не больше одного раза одновременно, и всегда ведёт к реальной форме редактирования.

### H5. «Отправить клиенту» срабатывает в один клик без предпросмотра/подтверждения
**Файл:** `js/page_order_card.jsx`, кнопка в `oc-actions`: `onClick={() => toast('Заказ отправлен клиенту', 'ok')}`.
**Ожидаемый результат:** оборачивай отправку в `ConfirmDialog` (минимум — «Вы уверены? Будет отправлено клиенту: …») или `Modal` с кратким предпросмотром того, что отправляется (например, последнее КП/документы), и кнопкой подтверждения внутри. Переиспользуй `ConfirmDialog` из `ui.jsx`.
**Приёмка:** прежде чем клиенту что-то фактически «отправляется», оператор видит, что именно, и подтверждает действие отдельным кликом.

## MEDIUM

### M1. Несогласованный скролл карты мест: аккордеон-шаг vs fullscreen-режим
**Файл:** `js/page_avia_picker.jsx`, `ApStep` (шаг 3, `SeatSelector`) vs `.ap-extras-ov`/`.ap-extras-scroll` (полноэкранный режим допуслуг).
**Ожидаемый результат:** если после H1 шаг 3 остаётся (или содержимое переехало в шаг 4) — содержимое раскрытого шага с картой мест должно скроллиться **внутри своего блока** (фиксированная max-height + `overflow-y:auto`), а не растягивать всю страницу аккордеона, аналогично тому, как уже сделано в `.ap-extras-scroll`.
**Приёмка:** раскрытие шага с картой мест не увеличивает высоту всей страницы сверх ~1.5 экрана.

### M2. Неструктурированная форма «Добавить пассажира»
**Файл:** `js/order_extras.jsx`, компонент `PassengerDrawer`.
**Ожидаемый результат:** разбить `form-grid` на 2 логические секции с подзаголовками (например, через тот же визуальный приём, что у `NewOrgDrawer` — секция «Расчётные счета» отделена заголовком): «Личные данные» (ФИО, дата рождения, гражданство) и «Документ и контакты» (тип/номер документа, срок действия, email, телефон, комментарий).
**Приёмка:** форма визуально делится на 2 блока вместо одной сетки полей.

### M3. Тайм-лимиты — только текст, нет визуального таймера
**Файлы:** `js/page_order_card.jsx` (`ScenarioCard`/`CalcLines`), `js/page_flights.jsx` (`FlightCard`, строка «Тайм-лимит … сегодня 18:00»).
**Ожидаемый результат:** там, где сейчас текст вида `style={{ color: 'var(--red)' }}>сегодня 18:00`, заменить на компактный визуально выделенный badge/Pill с иконкой `clock` (используй существующий `Pill tone="red"` или `amber` в зависимости от того, сколько времени осталось) — не обязательно живой JS-таймер с тикающими секундами, достаточно стабильно заметного визуального акцента, который не теряется в тексте.
**Приёмка:** тайм-лимит виден на карточке услуги как акцентный элемент, а не как часть обычного текста.

### M4. Нет явного мультивалютного индикатора в финансах заказа
**Файл:** `js/page_fulfillment.jsx`, `FinanceRegistry` (когда `scopeOrder` задан).
**Ожидаемый результат:** если у заказа операции в разных валютах одновременно — в блоке статистики (`STATS`/`grid-4`) показывать суммы с разбивкой по валюте, а не одним смешанным числом (или явно подписать валюту итога, если все суммы уже приведены к одной).
**Приёмка:** в карточке заказа с мультивалютными услугами видно, в какой валюте каждая сумма, без необходимости открывать каждую операцию.

### M5. Toast-уведомления перекрывают зону поиска/быстрых действий и накапливаются
**Файл:** `js/ui.jsx`, `ToastProvider`/`.toast-wrap` (CSS в `index.html`).
**Ожидаемый результат:** проверь текущее CSS-позиционирование `.toast-wrap` в `index.html` — сдвинуть так, чтобы тосты не перекрывали `.gtop-search`/`.gtop-actions` (например, опустить контейнер тостов ниже шапки `.gtop`, добавив `top` с отступом высоты `.gtop`, либо переместить в правый нижний угол). Также ограничить очередь видимых тостов (например, не больше 2 одновременно, остальные — в очередь) если такого лимита ещё нет в `ToastProvider`.
**Приёмка:** после серии быстрых действий (например, в мастере бронирования) тосты не закрывают поиск/колокольчик/переключатель роли дольше доли секунды.

### M6. Неочевидный путь «куда вносить документы клиента» после создания заказа
**Файл:** `js/page_orders.jsx`, `OrderCreateModal` (или его актуальный аналог — drawer создания заказа).
**Ожидаемый результат:** после успешного создания заказа и попадания в `OrderCard` (через `handleCreated`) — показать короткий onboarding-намёк (например, тост или подсказку в `TabParticipants`, если участников 0: «Добавьте участников и их документы здесь»), чтобы не заставлять нового оператора угадывать.
**Приёмка:** свежесозданный заказ без участников явно подсказывает, где добавить пассажиров/документы.

## Порядок работы

1. Сделай **Critical** (C1–C5) — это блокеры реальных сценариев.
2. Затем **High** (H1–H5).
3. Затем **Medium**, **Low** — по возможности одним проходом.
4. После каждого блока — короткий прогон в браузере (логин → нужный сценарий) и отчёт: что изменилось, как проверено, скриншот при необходимости.
5. Не трогай файлы `page_finance.jsx`/`page_docs.jsx`/мёртвый `OrderDetail` — они уже удалены в рамках предыдущей чистки, не восстанавливай их.
